== Краткое описание актуальности проблемы
Крупные компании и кадровые агентства ежедневно размещают сотни вакансий на отечественных платформах и внутренних системах подбора. При этом каждая вакансия может получать от десятков до тысяч откликов, значительная часть которых не соответствует заявленным требованиям (по уровню опыта, стеку технологий, формату занятости и т.д.).
В условиях ограниченных ресурсов рекрутинга (времени, экспертизы, скорости обработки) ручной разбор откликов становится узким местом.
В результате компании сталкиваются с неэффективной воронкой найма, где наличие большого количества данных не приводит к повышению качества решений.
Требуется разработать интеллектуальный сервис — ИИ-ассистент рекрутера, ориентированный на автоматизацию и повышение качества подбора персонала в условиях российского рынка труда.

== Краткая постановка задачи (с видом конечного результата и разделением на уровни: Базовый, Средний и Продвинутый уровень)
Необходимо разработать сервис ИИ-ассистента рекрутера. Основные функции ИИ-ассистента:
1. Анализировать тексты (txt, pdf, docx) описаний предлагаемых вакансий:
   - Выявлять ключевые сущности вакансий: требования к "хард-скиллам" и "софт-скиллам";
   - Выявлять требования к опыту/уровню владения навыками;
   - Выявлять прочие требования к кандидату: например, локация, формат работы (удаленный/офис/гибрид), часы работы.
2. Хранить в структурированном виде выявленные требования вакансий и необходимую метаинформацию к ним.
3. Анализировать тексты (txt, pdf, docx) резюме кандидатов:
   - Выявлять ключевые сущности резюме кандидата: "хард-скиллы" и "софт-скиллы";
   - Выявлять имеющийся опыт/уровень владения навыками;
   - Выявлять прочие существенные характеристики кандидата: например, локация, желаемый формат работы (удаленный/офис/гибрид), часы работы.
4. Хранить в структурированном виде выявленные характеристики кандидатов и необходимую метаинформацию к ним.
5. По запросу формировать:
   - Список первых N кандидатов по запрашиваемой вакансии. К каждому кандидату из результата должно идти развернутое текстовое описание с обоснованием, почему он подходит к вакансии и каких навыков ему не хватает (некритично для вакансии). Сортировка — от самого релевантного кандидата к менее релевантному. Кандидаты ниже порога 85% релевантности считаются не подходящими;
   - Список первых K вакансий, подходящих кандидату из запроса. К каждой вакансии из результата должно идти развернутое текстовое описание с обоснованием, почему она подходит кандидату и каких навыков ему для нее не хватает. Сортировка — от самой релевантной вакансии к менее релевантной. Вакансии ниже порога 85% релевантности считаются не подходящими;
   - Список первых X вакансий, по которым в базе нет подходящих кандидатов (т.е. все кандидаты имеют релевантность <=85%);
   - Список первых Y кандидатов, для которых в базе нет подходящих вакансий (т.е. все вакансии имеют релевантность <=85%).

= Базовый уровень:
1. Реализована минимально необходимая для функциональности архитектура.
2. Для хранения можно использовать обычные файлы без подключения СУБД.
3. Анализировать тексты вакансий и резюме только в txt-формате (кодировка UTF-8).
4. Взаимодействие с ИИ-ассистентом производится через терминал вызовом команды с передачей необходимых для работы параметров.
5. Вызов сервиса с параметром --help или -h выводит в терминале краткую справку по параметрам и правилам работы с сервисом.
6. Для демонстрации работы сервиса подготовлен набор команд (скриптов).
7. Система корректно работает при позитивных и негативных сценариях (не падает, даёт понятные сообщения об ошибках).
8. Проект доступен в git-репозитории (приватном).
9. Количественная оценка проверки тестовых резюме: верно оценена релевантность у 80%+ резюме = 20 баллов, 50-80% = 10 баллов, 0-50% = -20 баллов (штраф).

= Средний уровень:
1. Реализовано REST API (OpenAPI) для взаимодействия с сервисом.
2. Реализована аутентификация в сервисе ИИ-ассистента (логин/пароль или API-токен).
3. Присутствует Swagger-документация ко всем API сервиса с описанием структур запросов и ответов.
4. Анализ текстов из форматов .txt, .pdf, .docx.
5. Хранение сущностей реализовано в СУБД (реляционной или векторной — на выбор команды).
6. Поиск результатов производится с помощью запросов к СУБД (не полным перебором в памяти).
7. Выявление сущностей из текстов производится с помощью русскоязычных моделей ИИ (например, GigaChat).
8. Формирование сопроводительных текстов с обоснованиями производится с помощью русскоязычных моделей ИИ.
9. Модель ИИ при анализе и формировании ответов не должна "галлюцинировать" — запрещено придумывать отсутствующие в текстах факты или делать догадки.
10. Для демонстрации работы системы подготовлены запросы ко всем REST API (Swagger, curl, Postman и т.п. — на выбор команды).
11. Ни один вызов API не должен возвращать ошибку 500. Система стабильна при позитивных и негативных сценариях, возвращает понятные ответы даже при сбоях взаимодействия с ИИ-моделями.
12. Реализована вся функциональность базового уровня.

= Продвинутый уровень:
1. Вся функциональность среднего уровня.
2. Веб-интерфейс для рекрутера с аутентификацией (отдельный фронтенд или встроенный).
3. Упаковка проекта в контейнер (Dockerfile). Возможность запуска сервиса и зависимых сервисов с помощью Docker Compose.

= Дополнительные баллы:
1. Использование эмбеддингов и векторной СУБД в архитектуре сервиса, если это оправдано и улучшает качество поиска.
2. Адаптивная верстка веб-интерфейса, удобная на мониторе, планшете и смартфоне.

== Ограничения к стеку технологий (если применимо)
Выбор языка программирования, фреймворков и библиотек на усмотрение команды. Далее приведён список рекомендаций:
- Python: FastAPI (предпочтительно) / Flask / Django (наименее предпочтительно), pypdf, python-docx, gigachat, langchain, pandas;
- Java: Spring Boot (предпочтительно), Apache PDFBox, Apache POI, OpenCSV, Jackson, JPA/Hibernate;
- C#: ASP.NET Core (предпочтительно), iTextSharp (или QuestPDF), DocX, Newtonsoft.Json, Entity Framework Core;
- JavaScript / TypeScript: NestJS (предпочтительно) / Express.js;
- Фронтенд: React / Next.js / Vue.js / Angular;
- Контейнеризация: Docker, Docker Compose;
- Базы данных: PostgreSQL + pgvector / Chroma / Qdrant / LanceDB.

ИИ: GigaChat / GigaChain (https://developers.sber.ru/docs/ru/gigachain/tools/python/gigachat) — для личного пользования бесплатный тариф Freemium (https://developers.sber.ru/docs/ru/gigachat/tariffs/individual-tariffs), Transformers, LangChain.
Хранение кода: приватный репозиторий в GitVerse / GitHub / GitLab.

== Критерии оценки полученного результата
Технические показатели:
- Качество извлечения текста, отсутствие разрыва контекста при извлечении текста;
- Необходимое и достаточное извлечение сущностей из текста, отсутствие избыточности;
- Оптимальность архитектуры решения, возможность дальнейшего масштабирования решения без рефакторинга.

Пользовательский опыт:
- Презентация решения;
- Удобство взаимодействия с сервисом: логичные понятные параметры командной строки / REST API, интуитивный не перегруженный веб-интерфейс.

Разработка и развертывание:
- Документирование кода, чистота кода, отсутствие антипаттернов проектирования и разработки;
- Работа с git, проект доступен в приватном репозитории;
- Упаковка проекта в контейнер (Dockerfile). Возможность запуска сервиса и зависимых сервисов с помощью Docker Compose;
- Документация (README.md, Swagger).
